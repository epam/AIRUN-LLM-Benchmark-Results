────────────────────────────────────────────────────────
1. Analysis of the AngularJS application
────────────────────────────────────────────────────────
Main functional area
• A lightweight CMS “Page Editor”.  
• Lets an administrator create, edit, duplicate, delete and schedule “pages” that will later be rendered by the public-facing site.

Component / data flow
• REST factory – a thin wrapper around $resource that exposes CRUD endpoints (blocks, content, revisions, extras, tags, users, etc.).  
• Page factory – single in-memory object acting as a global cache for the page currently being edited.  
• Users factory – holds the currently logged-in user.  
• pageCtrl – orchestrates UI, business logic and persistence. It reads/writes Page, talks to REST and exposes data to page.html.  
• page.html – the form that a CMS user sees.

REST integration pattern
• Conventional RESTful URIs (api/content/:contentID, api/content/:contentID/revisions, …).  
• save() → POST (create), update() → PUT, delete() → DELETE.  
• Separate endpoints for child resources (extras, tags, requirements, etc.) to minimise payloads.

────────────────────────────────────────────────────────
2. User interaction patterns
────────────────────────────────────────────────────────
Visible UI elements
• Alert strip for “There is a newer unsaved local version”.  
• Toolbar (back arrow, title, close button).  
• Action bar: Delete, Duplicate, Save (switches to yes/no confirmation on delete).  
• Form fields
  – Page type selector (populated from themePages).  
  – Title + live character counter.  
  – Description + live character counter.  
  – Tag input with auto-complete (max 10 suggestions).  
  – URL slug editor (auto-generated while typing title until the user edits manually).  
  – Publish state radio buttons (Publish now / Draft / Schedule) with date-time picker when “Schedule” is chosen.

Typical user workflows
1. Create new page  
   • Route /new loads a blank Page object.  
   • User fills in form, presses “Save” → POST /api/content → POST /api/content/:id/revisions → optional POSTs for tags & extras → redirected to /:url.

2. Edit existing page  
   • Route /:url loads an existing Page (not shown in snippet).  
   • User changes fields, presses “Save” → PUT /api/content/:id → creates a new revision → replaces tags & extras.

3. Duplicate page  
   • “Duplicate” opens same save flow but forces a new URL.

4. Delete page  
   • Confirmation → DELETE cascade: content, revisions, extras, tags → redirect to /new.

Form validation and feedback
• Hard stops with translated error notifications when:  
  – Duplicate URL on “Duplicate” action.  
  – No page type selected.  
  – Blank or “new” URL.  
• Success/failure toasts translated via $translate and $rootScope.$broadcast('notify', …).

3. Extracted business requirements / objectives
────────────────────────────────────────────────────────
Business domain
• A multi-author Content Management System supporting Pages, Blocks, Files, Menus, Themes, Users, etc.

Key entities
• Page (aka “content”) – versioned, typed, taggable, schedulable.  
• Revision – immutable snapshot of a Page at save time.  
• Extra – arbitrary key/value meta data attached to page and to revision.  
• Tag – simple label, many-to-many with Pages.
